# Part I — Software Requirements Specification

## Vision
بناء نظام تخطيط موارد المؤسسات (ERP) متكامل ومتخصص لشركات **بيع وتوزيع المواد الغذائية بالجملة**. يهدف النظام إلى أتمتة العمليات من لحظة شراء البضائع وتخزينها، مروراً بإدارة الصلاحيات والتعبئة، ووصولاً إلى المبيعات، الفواتير، والتوصيل للعملاء (سوبرماركت، محلات تجزئة، إلخ).
* **الواجهة:** عربية بالكامل (RTL Support) بتصميم عصري وسهل الاستخدام للمستودعات والمبيعات.
* **الهدف:** تقليل الهدر في المواد الغذائية المنتهية الصلاحية، تحسين مسارات التوزيع، وضبط الحسابات بدقة.

## Business Rules
1. **إدارة الصلاحيات والتواريخ (Expiry & Batch Management):**
   * لا يمكن إدخال أي مادة غذائية للمخزن دون تسجيل (رقم التشغيلة Batch Number، وتاريخ الانتهاء Expiry Date).
   * يعتمد النظام قاعدة **FEFO (First Expired, First Out)** تلقائياً عند اقتراح الكميات للبيع.
2. **وحدات القياس المتعددة (Multi-UoM):**
   * دعم البيع والشراء بوحدات متعددة (طرد، كرتونة، كيس، حبة) مع نسب تحويل ثابتة (مثال: 1 كرتونة = 12 حبة).
3. **سياسات التسعير والخصومات:**
   * دعم أسعار مختلفة لنفس المنتج (سعر الجملة، سعر نصف الجملة، سعر التجزئة).
   * تطبيق حد ائتماني (Credit Limit) لكل عميل جملة لا يمكن تجاوزه عند البيع الآجل إلا بموافقة المدير.
4. **مرتجعات المواد الغذائية:**
   * تصنيف المرتجعات (صالح لإعادة البيع، تالف بسبب العميل، تالف بسبب النقل) مع معالجة محاسبية ومخزنية فورية.

## Functional Requirements
* **إدارة المخازن (Inventory):** جرد المخازن، التحويل بين المستودعات، تتبع التالف، وإشعارات قرب انتهاء الصلاحية.
* **إدارة المشتريات والموردين (Purchasing):** طلبات الشراء، فواتير المشتريات، حسابات الموردين، وتكاليف الشحن الإضافية.
* **إدارة المبيعات والعملاء (Sales):** عروض الأسعار، طلبات البيع، الفواتير الفورية والآجلة، وإدارة مناديب المبيعات.
* **الحسابات والمالية (Finance):** دليل الحسابات (Charts of Accounts)، قيود اليومية الآلية، سندات القبض والصرف، وضريبة القيمة المضافة (VAT).
* **التوزيع واللوجستيات (Fleet & Delivery):** تجهيز الطلبيات (Picking Lists)، تخطيط مسارات سيارات التوزيع، وتسليم الشحنات.

## Non-Functional Requirements
* **الأداء:** استجابة واجهة المستخدم في أقل من 1.5 ثانية للعمليات المعتادة.
* **التوافقية:** دعم كامل للغة العربية والاتجاه من اليمين إلى اليسار (RTL) في الواجهات والتقارير المطبوعة (PDF).
* **التوفر:** قدرة النظام على العمل أوفلاين جزئياً لمناديب المبيعات عبر تطبيق الويب وتزامن البيانات لاحقاً.

## Security
* تشفير كلمات المرور باستخدام `bcrypt` (أو خوارزمية حديثة مكافئة في بايثون).
* حماية جميع مسارات الـ API باستخدام JWT Tokens قصيرة الأجل مع Refresh Tokens.
* منع هجمات CSRF، SQL Injection، و XSS عبر استخدام حزم الأمان الافتراضية في FastAPI و React.

## Permissions (RBAC)
نظام صلاحيات صارم قائم على الأدوار (Role-Based Access Control):
* **مدير النظام (Admin):** صلاحيات مطلقة.
* **أمين المستودع (Storekeeper):** إدارة المخزون، الاستلام، وتجهيز الشحنات فقط.
* **موظف المبيعات/المناديب (Sales):** إنشاء فواتير البيع والتحصيل لعملائه فقط.
* **المحاسب (Accountant):** مراجعة القيود، الفواتير، التقارير المالية، والضرائب.

# Part II — Software Architecture

## Clean Architecture
يعتمد المشروع على بنية برمجية نظيفة لفصل المهام وتسهيل الصيانة والاختبار:

```
[ Domain (Models) ] <--- [ Application (Services/Use Cases) ] <--- [ Infrastructure (API/DB) ]
```

* **Domain Layer:** تحتوي على الكيانات الأساسية (Entities) وقواعد العمل الثابتة (Business Logic) الخالية من أي اعتمادات خارجية.
* **Application Layer:** تحتوي على الواجهات (Interfaces)، الخدمات (Services)، وحالات الاستخدام (Use Cases).
* **Infrastructure Layer:** تحتوي على إعدادات قاعدة البيانات (SQLAlchemy)، برامج تشغيل خارجية، والـ APIs (FastAPI).

## Folder Structure
يجب الالتزام بالهيكل التالي بدقة أثناء إنشاء المجلدات والملفات:

```
backend/
├── app/
│   ├── core/         # الإعدادات العامة، الأمن، الثوابت
│   ├── db/           # جلسة قاعدة البيانات، الهجرات (Alembic)
│   ├── domain/       # الكيانات النقية وقواعد العمل (Models)
│   ├── services/     # منطق الأعمال والخدمات (Use Cases)
│   ├── api/          # مسارات FastAPI والمخططات (Schemas/DTOs)
│   │   └── v1/       # إصدار الـ API الأول
│   └── tests/        # الاختبارات الأحادية والتكاملية
├── main.py           # نقطة انطلاق التطبيق
└── requirements.txt
frontend/
├── src/
│   ├── assets/       # الصور والخطوط العربية (مثل Tajawal)
│   ├── components/   # المكونات المشتركة وقابلة لإعادة الاستخدام
│   ├── context/      # إدارة الحالة العامة (Auth, Theme)
│   ├── hooks/        # الخطافات المخصصة (Custom Hooks)
│   ├── pages/        # صفحات النظام (المخازن، المبيعات، إلخ)
│   ├── services/     # الاتصال بالـ API (Axios client)
│   └── App.js
```

## Modules
يتكون النظام من الوحدات الأساسية التالية (يجب عزل كل وحدة في مجلد فرعي داخل الـ API والخدمات):
1. **Authentication (auth):** إدارة المستخدمين وتسجيل الدخول.
2. **Inventory (inventory):** الأصناف، المخازن، التشغيلات (Batches)، ووحدات القياس.
3. **Purchases (purchases):** الموردين، فواتير الشراء، والمدفوعات.
4. **Sales (sales):** العملاء، فواتير البيع، مرتجعات المبيعات، وعمولات المناديب.
5. **Accounting (accounting):** الحسابات، القيود، وميزان المراجعة.

## Database Design
* **قاعدة البيانات:** PostgreSQL.
* **العلاقات الأساسية:**
  * كل صنف (`Product`) له عدة تشغيلات (`ProductBatch`) لتتبع الصلاحية.
  * كل فاتورة بيع (`SalesInvoice`) ترتبط بعميل (`Customer`) ومندوب (`Salesman`) وتحتوي على عدة تفاصيل (`InvoiceLine`).
  * كل عملية مالية تولد قيود يومية تلقائية (`JournalEntry`) تحتوي على طرف مدين ودائن (`JournalItem`) لضمان القيد المزدوج.

## API Design
* استخدام معايير RESTful API.
* تسمية المسارات بالجمع وبأحرف صغيرة (مثال: `/api/v1/products/`).
* إرجاع استجابات موحدة ومفهومة بالصيغة التالية:
  ```json
  {
    "success": true,
    "data": {},
    "message": "تمت العملية بنجاح"
  }
  ```

## Services
كل عملية منطقية يتم عزلها في خدمة (Service class) مستقلة، على سبيل المثال:

* `InvoiceService.create_invoice()`: تتكفل بخصم المخزون حسب قاعدة FEFO، التحقق من حد الائتمان للعميل، وإنشاء قيد المحاسبة التلقائي في قاعدة البيانات ضمن Transaction (جلسة واحدة تضمن التراجع في حال الفشل).

# Part III — Coding Standards

## Python Rules
* الالتزام الصارم بـ PEP 8.
* استخدام الـ Type Hinting بشكل كامل في جميع الدوال والمتغيرات (مثال: `def get_item(id: int) -> ProductModel:`).
* استخدام فحص الأخطاء والتنسيق التلقائي عبر `ruff` أو `black`.

## FastAPI Rules
* استخدام الـ Dependency Injection الخاص بـ FastAPI لإدارة جلسات قاعدة البيانات `get_db` والتحقق من الصلاحيات.
* معالجة الأخطاء بشكل مركزي باستخدام `HTTPException` مخصصة ترجع رسائل واضحة بالعربية للمستخدم النهائي.
* كتابة توثيق لكل مسار باستخدام Docstrings ليظهر بشكل مرتب في `/docs` (Swagger UI).

## React Rules
* استخدام Functional Components مع React Hooks.
* استخدام مكتبة TailwindCSS للتنسيق مع إضافة خاصية `dir="rtl"` للوسم الرئيسي وتنسيق النصوص لتناسب واجهة عربية محترفة (استخدام خط "Tajawal" أو "Cairo").
* إدارة الحالة باستخدام React Context أو Redux Toolkit للعمليات المعقدة.
* **الجداول دائماً مقسّمة على صفحات (Pagination):** أي جدول بيانات في الواجهة يجب أن يعرض 15 عنصراً كحد أقصى في الصفحة الواحدة، لتجنب طول الصفحة وبطء الأداء. هذا مطبّق مركزياً في مكوّن `Table` المشترك (`frontend/src/components/Ui.jsx`)، فلا حاجة لإعادة تنفيذه في كل صفحة.
* **الألوان والتباين:** عند اختيار ألوان الشرائح/الحالات (Badges)، التزم بنمط خلفية فاتحة + نص غامق من نفس التدرج (مثال: `bg-emerald-100 text-emerald-800`)، وتجنّب النص الأبيض على خلفية فاتحة لضمان وضوح القراءة دائماً.

## SQLAlchemy Rules
* استخدام الـ Async Session لـ SQLAlchemy 2.0 لتحسين أداء النظام وتعامله مع الطلبات المتزامنة.
* تفعيل الـ Lazy Loading بحذر لعدم الوقوع في مشكلة (N+1 queries)، وتفضيل `selectinload` أو `joinedload` عند الحاجة لجلب البيانات المرتبطة.

## Naming Convention
* **Python/Backend:**
   * المجلدات والملفات: `snake_case`.
   * الفئات (Classes): `PascalCase`.
   * الدوال والمتغيرات: `snake_case`.
* **Frontend/React:**
   * المكونات والصفحات: `PascalCase` (مثال: `InvoiceDetail.jsx`).
   * الدوال والمتغيرات: `camelCase`.

## Git Rules
* رسائل الـ Commit يجب أن تتبع معايير (Conventional Commits)، مثال:
   * `feat(sales): add FEFO algorithm for batch selection`
   * `fix(auth): resolve JWT expiration token issue`

# Part IV — Claude Code Instructions

## Development Workflow
عند قيامك (يا Claude Code) بالعمل على هذا المشروع، اتبع الخطوات التالية بدقة:

1. **التحليل أولاً:** اقرأ المتطلبات البرمجية والملفات الحالية قبل كتابة أي كود جديد.
2. **التصميم المعماري:** تأكد من مواءمة أي إضافة برمجية مع بنية Clean Architecture المذكورة في الجزء الثاني.
3. **الكتابة التدريجية:** قم ببناء الواجهة الخلفية (الـ APIs والاختبارات)، وتأكد من عملها بنسبة 100% قبل الانتقال لبناء واجهة المستخدم المقابلة لها في React.

## AI Rules
* **لا تفترض أبداً:** إذا غابت عنك معلومة حول طبيعة العمليات التجارية لبيع المواد الغذائية بالجملة، اسأل المطور فوراً قبل المتابعة.
* **اللغة العربية في الواجهة:** جميع الرسائل الموجهة للمستخدم، نصوص الأزرار، عناوين الجداول، والتنبيهات يجب أن تُكتب بلغة عربية فصحى، واضحة وبسيطة.
* **التعليقات في الكود:** اكتب تعليقات توضيحية داخل الكود باللغة الإنجليزية للمطورين، لكن اشرح منطق العمليات المعقدة بوضوح.

## Module Generator
عندما يُطلب منك إنشاء وحدة (Module) جديدة (مثال: وحدة الموردين)، قم بتوليد الملفات بالترتيب التالي:

1. `domain/models/`: تعريف موديل قاعدة البيانات وجدول SQLAlchemy.
2. `api/schemas/`: تعريف مخططات Pydantic للإدخال والإخراج والتحقق من البيانات.
3. `services/`: بناء الـ Service الذي يحتوي على منطق العمليات وصياغة البيانات.
4. `api/v1/`: ربط الـ Router بـ FastAPI وتطبيق الصلاحيات.
5. `tests/`: كتابة اختبارات المسار للتأكد من السلامة البرمجية.

## Do & Don't
* **افعل (Do):**
   * استخدم العمليات البرمجية الآمنة (Database Transactions) لضمان عدم حفظ بيانات جزئية في حال فشل أي خطوة (مثال: عند حفظ الفاتورة، يجب أن ينجح خصم المخزون وحفظ القيود المالية معاً أو يفشلا معاً).
   * اهتم جداً بتجربة مستخدم المخازن (السرعة في إدخال البيانات عبر الكيبورد بدون الحاجة للماوس دائماً).
* **لا تفعل (Don't):**
   * لا تدمج كود قاعدة البيانات مباشرة في مسارات الـ API (الـ Controllers)؛ التزم بفصلها في طبقة الـ Services.
   * لا تغفل معالجة القيم العشرية (Decimals) الخاصة بالأسعار والكميات والمكاسب، لا تستخدم `float` أبداً للحسابات المالية؛ استخدم `Decimal` دائماً لتفادي مشاكل التقريب.

## Architecture Protection
* احرص على حماية البنية الهيكلية؛ إذا طلب المطور كتابة كود سريع يخالف معايير Clean Architecture، نبهه بلطف واقترح عليه الحل المعماري الأصح أولاً، ثم اتبع توجيهاته النهائية.

## Sequence Diagrams (Example: Create Sales Invoice with FEFO)

1. **User (UI):** يرسل طلب حفظ فاتورة مبيعات يحتوي على معرف العميل والمنتج والكمية المطلوبة.
2. **FastAPI Route:** يستقبل الطلب، ويتحقق من الـ JWT Token وصلاحية البائع.
3. **Invoice Service:**
   * يتحقق من حد ائتمان العميل (Credit Limit).
   * يستعلم من قاعدة البيانات عن المجموعات المتوفرة (`ProductBatches`) للمنتج المراد بيعه ويرتبها تصاعدياً حسب تاريخ الانتهاء (FEFO).
   * يخصم الكمية المطلوبة من التشغيلات الأقرب لانتهاء الصلاحية أولاً.
   * يسجل الأسطر المخصومة في تفاصيل الفاتورة (`InvoiceLines`).
   * ينشئ القيود المحاسبية الآلية (من حساب ذمم العملاء إلى حساب إيرادات المبيعات وحساب ضريبة المبيعات).
4. **Database:** يحفظ كافة الحركات السابقة في Transaction واحد ويقوم بعمل `Commit`.
5. **FastAPI Route:** يعيد استجابة نجاح مع ملف الفاتورة الجاهز للطباعة.

## Future Roadmap
* **المرحلة الأولى:** بناء النواة (المستودعات، المشتريات، المبيعات الأساسية والـ API).
* **المرحلة الثانية:** بناء النظام المالي وقيود اليومية التلقائية وإعداد التقارير الضريبية والمالية.
* **المرحلة الثالثة:** تطوير لوحة تحكم ذكية (Dashboard) للمبيعات والمخزون الراكد وتطبيق خاص بالمناديب يعمل بدون إنترنت (PWA Offline Sync).

## Index
* **FEFO:** First Expired, First Out - الوارد أولاً منتهي الصلاحية أولاً، يُصرف أولاً.
* **UoM:** Unit of Measure - وحدة القياس (طرد، كرتونة، كجم، حبة).
* **Batch:** التشغيلة أو رقم الدفعة الإنتاجية للمادة الغذائية لتسهيل تتبعها وصلاحيتها.
* **RBAC:** Role-Based Access Control - التحكم في الوصول القائم على أدوار المستخدمين.
